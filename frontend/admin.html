Order State Machine
Every order follows this pipeline. Each transition triggers downstream actions: stock deduction, SMS, and M-Pesa events.
📋 Order State Machine
Click any stage to see what it triggers
Stock Auto-Deduction
When an order transitions pending → paid, the backend runs a single DB transaction to decrement each component. For packages, it queries package_items and deducts each sub-component individually.
Simulate: Business Pro 10kW Order
☀️ MonoPerc 400W Panel ×2545→20
🔋 LiFePO4 100Ah Battery ×43→⚠️ 0
⚡ 3-Phase Inverter 10kW ×16→5
🔧 Mounting Hardware ×1 set12→11
Wrapped in BEGIN ... COMMIT — if any deduction fails, the entire transaction rolls back (ON DELETE RESTRICT).
Manual Payment Override
For corporate clients and schools paying via Paybill Bank Transfer, RTGS, or cheque. This bypasses the Daraja callback but triggers the same downstream logic: stock deduction + success SMS.
💳 Mark Order as Manually Paid

💡 Design Decisions
JSONB specifications: Batteries track Ah/DoD, panels track W/efficiency — different schemas per product type without separate tables.
ON DELETE RESTRICT: Prevents deleting a product that's tied to a live package — avoids breaking the catalog.
mpesa_checkout_id: Unique index for O(1) lookup when Safaricom's callback hits /api/v1/mpesa/callback.
price_at_purchase: Snapshot price at order time — price changes later never corrupt historical order records.
